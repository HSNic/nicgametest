# /update-changelog — 版本異動紀錄雙檔同步補寫

對應 `Lineage/加掛版/docs/版本異動紀錄/` 底下兩份檔案各自檔頭寫的「維護方式」。**兩份都要補,寫在各自檔案最上面(新的在前)**,內容分工不同,不要互相抄。

## 步驟

1. **蒐集這次改動的事實**(不確定就去問使用者或查 git,不要編):
   - `version.json` 的 `build` 值、對應的 commit hash(`git log -1 --format=%H`,還沒 commit 就寫「待補」)。
   - 改了哪些檔案、每個改動「為什麼」(根因/需求來源)。
   - 測試結果:跑過的指令(`smoke-hooks.mjs` 等)、Playwright/實機驗證範圍與結果、**有沒有做過的部分要老實寫清楚**(例如只測過假流程、沒連真後端),不要含糊帶過當作已驗證。

2. **技術版**(`版本異動紀錄_技術版.md`)照這個格式插在檔案最上面(標題列下方、第一個 `---` 分隔線之後):
   ```markdown
   ## <日期> build <build 值>(commit `<簡短hash>`)

   **類型:** 小版本(外掛/樣式調整) 或 大版本(同步原作者新版)

   **改了哪些檔案、為什麼:**
   - 條列,每項講清楚根因/需求來源

   **測試結果:**
   - smoke test 結果、手動測試結果(誠實標註測試範圍,沒測到的部分要寫出來)
   ```

3. **玩家版**(`版本異動紀錄_玩家版.md`)照這個格式插在最上面:
   ```markdown
   ## <日期> <小版本 / 大版本>(<一句話主題>)

   - 條列,每項一句話講清楚玩家會感受到的變化,不放技術細節、不放檔名/函式名
   ```
   - 白話文,程式小白看得懂;只講「玩起來有什麼不一樣」。
   - 有安全性/資料風險的提醒(例如「換瀏覽器會讀不到配對碼」)要用 ⚠️ 標出來。

4. **回報使用者**補了哪兩筆、放在哪個檔案,讓他能檢視文字是否合適。

## 判準

- 兩份檔案格式不同(技術版重過程與根因,玩家版只講感受),不要把技術細節貼進玩家版。
- 如果這次改動同時涉及本體同步(大版本)與外掛調整(小版本),照現有慣例可以合併成一筆或分兩筆寫,依實際 commit 拆分情況決定,參考 `版本異動紀錄_技術版.md` 既有寫法(同一天多筆 commit 通常各自一筆)。
