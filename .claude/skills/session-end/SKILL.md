# /session-end — 換 session 前收尾

目的:讓下一個 session(或下一次對話)只要跑 `/handoff` 就能接手,不必回來問「上一輪到底做完了沒」。**這是 `/handoff` 的反向 skill**——`/handoff` 是開場讀,這支是收尾寫。

## 步驟

1. **確認這次改動都乾淨落地**:
   ```bash
   git status --short --branch
   git log origin/main..HEAD --oneline   # 有沒有還沒 push 的 commit
   ```
   - 有未 commit 的改動:確認是要收尾進 commit,還是刻意留著(問使用者)。
   - 有未 push 的 commit:提醒使用者,不要假設他知道。

2. **如果這次動了程式碼**(`afk-*.js`/`index.html`/`js`/`css`):
   - 確認改動的外掛 `?v=` 已 bump、`node scripts/stamp-sw-version.mjs` 跑過、`node scripts/smoke-hooks.mjs` 通過。
   - 沒做完就直接跑 `/prepush` 補齊,而不是收尾時才發現漏了。

3. **如果這次的改動還沒寫進版本異動紀錄**(`Lineage/加掛版/docs/版本異動紀錄/` 兩份檔案最上面沒有對應這批改動的條目):
   - 跑 `/update-changelog` 補上。

4. **如果這批改動是「階段性成果、使用者已確認上線沒問題」,但還沒打 tag/Release**:
   - 跑 `/release` 走完 tag + 交接使用者 push/開 Release 的流程。
   - 不確定算不算「階段性成果」就直接問使用者,不要自己臆測。

5. **更新交接紀錄**(`Lineage/加掛版/docs/交接與接手/交接紀錄_YYYYMMDD.md`):
   - **同一天內的收尾** → 直接編輯當天那份既有檔案,把最新狀態寫上去(不必整篇重寫,更新「目前狀態」段落 + 補上這一輪做了什麼)。
   - **跨到新的一天** → 新開一份 `交接紀錄_YYYYMMDD.md`(當天日期),開頭註明「取代 `交接紀錄_<前一份日期>.md` 作為最新快照」。
   - 內容至少要包含:
     - 目前工作資料夾(固定 `idle-lineage-class`,不必再寫流水號)
     - git 狀態(是否乾淨、最新 commit、有沒有 push)
     - `version.json` 的 build 值、有沒有打 tag/Release
     - 這一輪(或這幾輪)完成了什麼、測試到什麼程度(誠實寫,沒測到的部分要講清楚,別讓下個 session 誤以為都測過)
     - 已知但先不處理的問題、待辦事項
     - 有沒有這次新學到、值得讓下個 session 知道的規則/教訓

6. **如果第 5 步開了新的交接紀錄檔案**:
   - 回頭更新 `Lineage/加掛版/docs/INDEX.md`「交接與接手」那一類的表格:把新檔案標為「**最新**」,原本標最新的那份改標「較早的現況快照(歷史參考)」(`README.md` 已不再列文件清單,以 `INDEX.md` 為唯一索引)。

7. **如果這次對話中踩到新坑、或使用者糾正/確認了某個非顯而易見的做法**:
   - 照 `CLAUDE.md` 既有慣例,找最貼近的章節補一條(標題一句話講結論、內文寫「為什麼會漏/錯 + 怎麼偵測或避免 + 判準」,附案例)。
   - 補完在回覆裡一句話講清楚補了哪條、補在哪,讓使用者能否決,不要默默塞進去。

8. **回報使用者一份簡短總結**(不是把交接紀錄全文貼出來):這輪做了什麼、有沒有東西還沒 push/沒打 tag、下一個 session 要注意什麼。

## 判準

- 不是每次小改動都要走完整套(尤其第 4 步打 tag)——只有使用者明確收尾/準備結束對話,或改動已是一個完整段落時才需要。改到一半就被要求「先收尾」時,如實記錄「這輪未完成、卡在哪」,不要為了收尾硬把半成品寫成「已完成」。
- 第 5-6 步的交接紀錄/INDEX.md 更新**每次收尾都要做**,這是讓下一個 session 能接手的最低限度,不可省略。
