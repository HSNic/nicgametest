# /handoff — 接手現況快照(精簡版)

目的:**只讀「現在還有效」的兩份文件**,不要每次都把 `Lineage/加掛版/docs/` 底下全部歷史文件讀一輪——那些是給「查特定歷史細節」用的參考資料,不是每次接手的必讀項目。

## 步驟

1. **讀 `Lineage/加掛版/docs/README.md`**,只看「目前文件清單」表格——**用表格裡標「最新」的那份交接紀錄檔名**,不要憑印象/猜日期去讀舊檔(README 才是權威索引,舊的交接紀錄檔名不會自己消失,但已被取代)。
2. **只讀那一份最新交接紀錄**(通常是 `交接與接手/交接紀錄_YYYYMMDD.md`),重點抓:
   - 目前工作資料夾路徑
   - 目前 git 狀態、最新 commit、`version.json` 的 build 值
   - 這一輪/上一輪做了什麼(只抓結論與是否已上線,不必複誦每個技術細節)
   - 有沒有「已知但先不處理的小問題」或「待補的 tag/Release」
3. **核對現況跟文件描述是否一致**(這步不能省,文件可能落後於實際狀態):
   ```bash
   cd <交接紀錄裡寫的工作資料夾路徑>
   git status --short --branch && git log --oneline -3 && cat version.json
   ```
   若對不上,以實際 git/version.json 為準,回報使用者哪裡不一致。
4. **只有在使用者接著問到具體歷史細節**(例如「雲端同步當初怎麼設計的」「某次同步差異分析內容」)**時**,才照 README 的分類表(交接與接手/同步/版本異動紀錄/發布上架/風險與外掛)去讀對應那一份,不要預先讀。

## 不要做的事

- 不要一次把 `docs/` 底下所有 md/csv 讀完再回報——那是「深度覆核」場景才需要的用量,平常接手不必。
- 不要複誦交接紀錄全文給使用者看,用「目前狀態三句話 + 待辦」的摘要回報就好,使用者要細節會再問。
