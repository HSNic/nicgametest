## 建議維護方式

### 方案 A:保守接手,先文件化再改

適合目前狀態。先保留快照,補文件、建立 Git、做第一次基準 commit。優點是風險最低;缺點是還不能立即吃上游更新。

### 方案 B:建立自己的 fork,以本快照為基準

適合要長期接手。建立自己的 GitHub repo,把目前快照作為初始 commit,再設定 upstream 指向 `pp771007/idle-lineage-class` 或原作者來源。優點是更新、回滾、比較最清楚;缺點是第一次整理較花時間。

### 方案 C:重新 clone 上游,再套本機差異

適合懷疑本機快照不完整時。優點是能確認完整歷史;缺點是需要仔細處理本機檔案與上游差異。

## 建議下一步

1. 在此快照資料夾內初始化 Git,建立「接手基準」commit。
2. 把 `docs/` 文件一起 commit,作為接手紀錄。
3. 建立自己的遠端 repo,避免只靠本機資料夾保存。
4. 第一次正式修改前,先跑一次本機靜態伺服器與 smoke test。
5. 每次改 `afk-*.js`,同步更新 `index.html` 中對應 `?v=`、`sw.js`、`version.json`。
