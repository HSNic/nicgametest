# Console 問題根因分析(2026-07-14)

> 對應任務:`Lineage/待辦-ClaudeCode/交接待辦/2026-07-14_Console問題修正代辦建議書(SW-Cache錯誤與動畫404).md`

## 問題 A:Service Worker Cache 錯誤(`AbortError: Failed to execute 'keys' on 'Cache': operation too large`)

### 根因

`sw.js` 的 `reconcileAnim()`(怪物動畫幀逐怪對帳)每次被呼叫時,**都會先對整個圖桶(`IMG_CACHE`)跑一次 `await cache.keys()`**,把所有已快取的檔案分組成「哪個怪資料夾有哪些檔案」,才能判斷要清掉哪些過期怪的動畫幀。

問題是:本專案的 `assets/anim/` 資料夾一款就有將近 9 萬個檔案(493 隻怪),全部 `assets/` 加起來超過 11 萬個檔案。玩家玩得越久、看過的怪越多,圖桶裡累積的快取項目就越多。當項目數大到一定程度,瀏覽器(Chromium)對 `Cache.keys()` 這種「一次列出全部項目」的操作有內部限制,超過限制就直接丟出 `AbortError: operation too large`,而且**這個呼叫沒有包 try/catch**,錯誤就變成 Console 裡「Uncaught (in promise)」的例外。

### 額外發現(意料之外的第二個問題,同一支函式裡)

深入看程式碼時,還發現 `reconcileAnim()` 判斷「要不要清除某隻怪」用的比對邏輯,期待收到的資料格式是 `[資料夾, sha]` 這種陣列(tuple),但實際上 `scripts/sync-upstream.mjs`/`scripts/sync-local-upstream.mjs` 產生的 `anim-manifest.json`/`assets-manifest.json` 一直都是 `{path, sha, size}` 這種物件格式(不是陣列)。

這代表:**這個「怪物動畫逐怪清除舊快取」的功能,從一開始就沒有真的運作過**——因為格式對不上,程式會把每一筆manifest資料都判斷成「沒有 sha 可比對」,直接跳過,永遠不會真的清除任何過期動畫。這件事一直沒被發現,是因為它「靜靜地不運作」,不會跳錯誤訊息(跟本次的 AbortError 崩潰是兩回事,但同一支函式)。

### 修正方式

- 把「先掃整個圖桶再分組」的做法,改成**每隻怪自己記自己抓過哪些檔案**(存在圖桶裡一個很小的專屬索引,不是整桶),清除時只讀寫這隻怪自己的小索引,不再需要對整桶喊 `cache.keys()`。
- 對「這次修復上線前」就已經快取好、還沒有專屬索引的舊資料,保留一次性的「舊快取遷移」機制(還是用 `cache.keys()`,但只在真的需要時才觸發,且整段包 try/catch——真的因為快取量太大掃不動,就放棄這次清除,不會讓遊戲跳錯,下次該怪動畫被重新抓取時會自動建立新索引)。
- 順便修正上面發現的資料格式問題,讓「逐怪清除舊動畫」這個功能真的開始運作(不是本次任務要求的,但既然是同一支函式的根因,一併修正)。
- 兩個 `message` 事件的 reconcile 呼叫都補上 `.catch()`,任何 Cache API 錯誤都只印警告、不會變成未捕捉的例外。

### 實際證據

- `sw.js` 原本第 167 行:`for (const req of await cache.keys())`,呼叫前後都沒有 try/catch。
- `anim-manifest.json` 實際內容範例:`{"dir":"assets/anim/不幸的幻象眼魔","name":"不幸的幻象眼魔","files":103,"size":527472,"sha":"..."}`——是物件不是陣列。
- `sw.js` 原本的 `manifestEntries()`/`reconcileAnim()` 判斷式 `Array.isArray(e)` 對這種物件恆為 false。

### 影響範圍

只影響 Service Worker(`sw.js`)的背景快取整理邏輯,不影響遊戲本身的讀取/存檔/戰鬥;修正前,圖桶越大越容易在背景丟出未捕捉的錯誤(但不影響畫面正常運作,純粹是 Console 噪音+背景整理失敗);修正後,消除這個錯誤來源,且動畫舊快取的清除功能開始真正運作。

### 未採用方案與理由

- **方案:整個圖桶改用「限制項目數 + LRU 淘汰」**(任務書提到的方案1):評估後不採用,因為玩家的圖片需求量本來就大(戰鬥怪物種類多),硬性上限容易造成「剛看過的怪動畫又被清掉、下次還要重抓」,體驗變差;根因是「怎麼找出要清哪些項目」的做法太貴,不是「桶子太大本身有問題」,所以改成逐怪小索引更對症。
- **方案:整桶按版本 bump 全部清空重建**:不採用,理由與專案既有設計一致(見 CLAUDE.md:圖桶固定桶名、不整桶倒掉,避免玩家重新下載整包 30MB+ 圖片)。

## 問題 B:動畫資源大量 404 Not Found

### 根因(結論:這不是bug,是原作者自己的設計)

深入讀 `js/09-vfx-render.js` 後發現,動畫幀數的偵測方式是**原作者自己刻意設計的探測機制**:遊戲沒有一份「每隻怪每個動作有幾幀」的清單,而是**逐號嘗試載入圖片(`idle_0.png`、`idle_1.png`、`idle_2.png`……),直到某一張 404 失敗,就代表這個動作到上一張為止**(`_mobAnimProbe()`/`probeSeq()`,程式裡的註解直接寫「探測逐號載到 404 即止」)。

也就是說:**每一隻怪、每一種動作,天生就會有「最後嘗試的那一張」是 404——這是設計上必然會發生的,不是缺檔案或路徑寫錯**。

### 這不是「無限重試」

程式有 `_mobAnimCache[name]` 做每隻怪的快取:同一隻怪在同一次遊戲過程中,只會探測一次(第一次遇到時做完整套探測、記住結果),之後所有出現都直接讀快取,**不會重複發出請求**。所以 Console 看到的 404 數量,等於「玩家這次遊戲總共遇過幾隻不同的怪(以及牠有幾種動作/幾層特效)」,不是無窮迴圈式的重複要求。

### 驗證

用無頭瀏覽器載入首頁(尚未進入遊戲、只有創角/登入畫面的動畫),確實看到約 10 筆 404(對應創角畫面的立繪/背景動畫探測),數量與探測邏輯預期的「每個動作一次到底的最後一張」吻合,重新整理同一頁面不會讓 404 數量疊加成兩倍(代表沒有重複探測)。

### 建議處理方式

因為這是原作者的正常設計、且已有防重複探測的快取機制,**不建議修改 `js/09-vfx-render.js` 的探測邏輯**(改了容易在「作者到底幾幀」判斷上出錯,且違反「不改本體」原則,益處只是讓 Console 好看一點,風險與效益不成比例)。

若還是想讓 Console 乾淨一點(純粹美觀,不影響任何遊戲功能),可行方向是:另外寫一支 `afk-*.js`,離線先掃描 `assets/anim/` 資料夾實際檔案數量、產生一份「每隻怪每個動作有幾幀」的清單,包住 `_mobAnimProbe()`,直接照清單建立幀陣列、不透過瀏覽器發送任何一次注定失敗的請求。**這件事這次沒有做**,因為:①效益純粹是主觀的「Console 好不好看」,不影響任何玩家體驗;②要完全複製原作探測邏輯的所有細節(影子層/武器層/第二武器層/技能特效層,各自不同前綴與最少幀數規則)才能保證行為完全一致,出錯風險不低;③已確認不是任何實際問題的根因。若你希望我還是做這個「純粹讓 Console 乾淨」的優化,請告訴我,我會另外抓時間仔細做。
